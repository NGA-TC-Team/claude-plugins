# NextGenAI AX 컨설팅 방법론

## 개요

NextGenAI의 AX(AI Transformation) 프로젝트는 고객사의 실제 업무 프로세스를 기반으로 AI 자동화를 설계·구축·정착시키는 실행 중심 컨설팅이다. 교육이 아닌 실행, 이론이 아닌 현장 적용이 핵심이다.

본 방법론은 **70~80%의 표준 기준선**이며, 나머지 20~30%는 PM 판단에 따라 조정 가능하다.

## 핵심 프레임워크: 4단계 구조

```
세일즈(무료 진단 1회) → 진단(2~3주) → 빌드(4~12주) → 확산(내재화)
```

### 1단계: 세일즈 (무료 진단 1회)

목표: 고객사 현황 파악 및 계약 전환

핵심 활동:
- 고객사 기본정보 리서치 (업종/조직/툴 스택/C-레벨 성향/매출규모)
- 사전 설문 발송 (AI 도입 사전진단 + AI 툴 활용도 진단)
- 무료 진단 미팅 운영 (60~90분): 트렌드 공유 → 설문 인사이트 → 병목점 파악 → 범위 합의 → 가격 안내

산출물: **Sales Brief 1-pager**
- 고객 목표 / Top Pain / 자동화 후보 Top5
- AI 리터러시 레벨링(초안)
- C-레벨 의지
- 다음 단계(진단 착수 범위/기간)

### 2단계: 진단 (2~3주)

목표: **Build팀이 추가 인터뷰 없이 바로 설계에 들어갈 수 있는 상태를 만드는 것**

핵심 활동:
- **킥오프(Week 0~1)**: 대상 부서 1~2개 확정, 핵심 태스크 후보 5~7개 수집, 인터뷰 일정 확정
- **정보 수집(Week 1~2)**: 업무 프로세스 인터뷰(반드시 화면 공유), 화면 녹화 확보, 데이터 소스 정의, 사용 툴/API 점검
- **AS-IS 문서화(Week 2)**: Step 단위 분해, 의사결정 지점, 승인 대기 구간, 예외 처리, 재작업 발생 지점, 평귰 반복 횟수
- **자동화 후보 3~5개 확정**: 구체 단위로 정의 (입력/처리/출력 명시)

산출물:
1. AS-IS 워크플로우 문서 (스텝 단위, 데이터 흐름, 승인 구조)
2. 파일/권한 맵 (파일 위치, 템플릿 위치, API 가능 여부, 권한 보유자)
3. 자동화 후보 3~5개 (문제 정의, 입력/출력 데이터, 예상 효과, 제약사항)
4. 제외 항목 명시 (범위 과도, API 불가, ROI 낮음)
5. Build팀 인계서

### 3단계: 빌드 (4~12주)

목표: 자동화 PoC 구축 및 현장 적용

핵심 활동:
- **착수(Week 1)**: 진단팀 인계서 리뷰, 스코프 재검증, 자동화 우선순위 재조정(난이도/효과/권한 기준)
- **교육(Week 1~2)**: AI 리터러시 + AI Thinking Process, "사용법"이 아닌 "업무에 적용하는 습관" 만들기
- **PoC 제작(Week 2~6)**: 표준 툴 N8N 기반, MVP → 현장 테스트 → 개선 루프, 확장 가능한 형태로 문서화
- **확산(Week 6~12)**: Champion 선정/육성, 운영 매뉴얼/재사용 템플릿 제공, 고도화 로드맵 제안

산출물:
1. 자동화 워크플로우 (운영 가능한 형태)
2. 운영 문서 (권한/에러 대응/주간·월간 보고)
3. 교육 자료/실습 예제
4. 성과 리포트 (전후 비교) + 확산 로드맵

### 통합 자동화 구축 프로세스 (8단계 Framework)

1. 업무 흐름 진단 (AS-IS 분석)
2. 자동화 니즈 정의 (Pain Point 도출)
3. 데이터·툴 진단 (API/권한 점검)
4. Agent 솔루션 설계 (Workflow 설계)
5. 자동화 MVP 구축 (PoC 제작)
6. 현장 테스트 (실사용 테스트)
7. 문서화·교육·인수인계
8. 지속 운영 및 고도화 (TO-BE 정착)

## 역할 정의

- **AX 세일즈팀**: 무료 진단 운영 / 계약 전환 / 기대치 정렬
- **AX 진단팀**: 2~3주 정밀 분석 / AS-IS 문서화 / PoC 스코프 확정 / 빌드팀 인계
- **AX Build팀**: 교육·PoC 제작·현장 적용 / 우선순위 재조정 / Champion 육성 / 확산

## 5대 원칙

1. **말로만 듣고 설계하지 않는다** — 반드시 실제 업무 흐름을 기반으로 AS-IS를 확인한다
2. **즉답이 어려운 사항은 신중하게 판단한다** — "검토 후 명확히 답변" 원칙 유지
3. **실행은 작은 단위로 나누어 검증한다** — 2주 내외의 반복 검증 구조
4. **기술·권한·보안 검토를 선행한다** — API/접근 권한/보안 정책 확인 없이 설계하지 않는다
5. **문서화는 프로젝트 자산으로 남긴다** — 고객사 공유용 기준으로 정리

## AI 리터러시 레벨링

설문 기반으로 3단계 분류:
- **Level 1**: AI 기초 활용 미흡
- **Level 2**: 도구 사용 경험 보유
- **Level 3**: 자동화·심화 모델 활용 가능

## 자동화 후보 선정 기준

1. 가장 귀찮고 반복적인 업무 3개 우선 선정
2. 자동화 시 의미 있는 변화 기준으로 우선순위: 고객 경��� 변화 / 리드 타임 단축 / 매출·리스크 영향
3. 기술 검증: 사용 툴 API 여부, Webhook 가능 여부, 접근 권한, 보안 정책
4. 구체 단위로 정의 (너무 넓은 범위 금지)

잘못된 예: "게시물 제작 자동화" (너무 넓음)
올바른 예: "수정사항 자동 취합 시스템 구축" (입력: 이미지블 수정 댓글 → 처리: 댓글 자동 수집 → 출력: 수정 목록 정리 문서)
